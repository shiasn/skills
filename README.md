# Skills 仓库

本仓库包含用于开发工作流的各种技能（Skills）。每个技能都是一个独立的指南，用于在特定场景下指导工作。

## 技能列表

### 流程类技能

#### brainstorming
**描述：** 将想法转化为完整的设计和规范。通过自然协作对话，理解项目上下文，逐步细化想法，提出多种方案，最终呈现设计文档。

**触发时机：** 在任何创意工作之前使用 - 创建功能、构建组件、添加功能或修改行为之前。必须先探索用户意图、需求和设计，再进行实现。

---

#### systematic-debugging
**描述：** 系统化调试方法。在尝试修复之前始终找到根本原因。症状修复是失败。

**触发时机：** ⚠️ **强制要求**：在遇到任何 bug、测试失败或意外行为时，在提出修复之前使用。用于测试失败、生产中的 bug、意外行为、性能问题、构建失败、集成问题。

---

#### development-workflow
**描述：** 协调设计、实现、测试和文档到统一的开发流程中。遵循迭代开发周期：设计 → 实现 → 改进 → 文档 → 审查。

**触发时机：** 开始新功能或重大变更、处理需要设计思考的 bug、任何跨越多次提交的工作、协调头脑风暴、TDD 和文档时使用。不适用于简单的一行修复、紧急热修复或一次性原型。

---

#### using-superpowers
**描述：** 建立如何查找和使用技能。在开始任何对话时使用。要求在包括澄清问题在内的任何响应之前调用 Skill 工具。

**触发时机：** ⚠️ **强制要求**：在开始任何对话时使用。如果认为有 1% 的机会某个技能可能适用于你正在做的事情，你必须调用该技能。这不是可选的。

---

### 前置检查类技能

#### checking-auto-imports
**描述：** 检查构建工具配置（rsbuild/vite/rslib/webpack）中的自动导入插件，并记录自动导入的内容。自动导入适用于值，不适用于类型。

**触发时机：** ⚠️ **强制要求**：在任何 JavaScript/TypeScript 项目任务之前使用。在开始任何 JS/TS 代码更改、添加新组件或文件、修改现有代码、调试导入相关问题、重构代码之前必须先执行。

---

#### react-19-changes
**描述：** 记录 React 19 相对于 React 18 的变更，包括 ref 作为 prop、forwardRef 弃用、Context 简化、Actions 和其他破坏性变更。

**触发时机：** ⚠️ **强制要求**：在编写 React 代码之前使用。在编写 React 组件、使用 React hooks、在组件之间传递 ref、使用 Context API、编写表单处理代码或任何 React 相关代码更改之前必须阅读。

---

#### zod-4-changes
**描述：** 记录 Zod 4.x 相对于 Zod 3.x 的变更，包括性能改进、API 变更（z.email()、toJSONSchema()）、Zod Mini 和其他破坏性变更。

**触发时机：** ⚠️ **强制要求**：在编写任何 Zod 验证代码之前使用。在编写 Zod 模式、使用 Zod 验证、定义数据验证规则、处理表单验证或任何 Zod 相关代码更改之前必须阅读。

---

### 实施类技能

#### frontend-design
**描述：** 创建独特、生产级的前端界面，具有高质量设计。生成创意、精美的代码和 UI 设计，避免通用 AI 美学。

**触发时机：** 用户要求构建 Web 组件、页面、工件、海报或应用程序时使用（包括网站、登录页、仪表板、React 组件、HTML/CSS 布局，或美化任何 Web UI）。

---

#### writing-plans
**描述：** 为多步骤任务编写全面的实施计划。假设工程师对我们的代码库零上下文且品味可疑。记录他们需要知道的一切：每个任务要接触哪些文件、代码、测试、他们可能需要检查的文档、如何测试。

**触发时机：** 当你有多步骤任务的规范或需求时，在接触代码之前使用。用于编写全面的实施计划，假设工程师对我们的代码库零上下文。

---

#### executing-plans
**描述：** 在单独会话中执行书面实施计划，带审查检查点。加载计划，批判性审查，批量执行任务，在批次之间报告以供审查。

**触发时机：** 当你有书面实施计划要在单独会话中执行，需要审查检查点时使用。适用于需要与架构师审查的批量执行。

---

#### subagent-driven-development
**描述：** 在当前会话中执行实施计划，每个任务分派新的子代理，每个任务后进行两阶段审查：首先规范合规性审查，然后代码质量审查。

**触发时机：** 当执行具有独立任务的实施计划，在当前会话中工作时使用。适用于任务大多独立、可以并行工作、不需要共享状态或顺序依赖的情况。

---

#### typescript-advanced-types
**描述：** 掌握 TypeScript 的高级类型系统，包括泛型、条件类型、映射类型、模板字面量类型和工具类型，用于构建类型安全的应用程序。

**触发时机：** 构建类型安全的库或框架、创建可重用的泛型组件、实现复杂的类型推断逻辑、设计类型安全的 API 客户端、构建表单验证系统、创建强类型配置对象、实现类型安全的状态管理、将 JavaScript 代码库迁移到 TypeScript 时使用。

---

### 质量保证类技能

#### verification-before-completion
**描述：** 在声称工作完成、修复或通过之前使用。要求在做出任何成功声明之前运行验证命令并确认输出。证据先于断言，始终如此。

**触发时机：** 在声称工作完成、修复或通过之前，在提交或创建 PR 之前使用。要求运行验证命令并确认输出，然后才能做出任何成功声明。

---

#### requesting-code-review
**描述：** 分派代码审查者子代理，在问题级联之前发现问题。早期审查，经常审查。

**触发时机：** 完成每个任务后（在子代理驱动开发中）、完成主要功能后、合并到 main 之前必须使用。可选但有价值：卡住时（新视角）、重构之前（基线检查）、修复复杂 bug 后。

---

#### receiving-code-review
**描述：** 接收代码审查反馈时使用，在实施建议之前，特别是如果反馈似乎不清楚或技术上可疑时。需要技术严谨性和验证，而不是表面同意或盲目实施。

**触发时机：** 接收代码审查反馈时，在实施建议之前使用，特别是如果反馈似乎不清楚或技术上可疑时。需要技术严谨性和验证，而不是表面同意或盲目实施。

---

### 工具类技能

#### browser
**描述：** 使用 Chrome DevTools Protocol (CDP) 进行浏览器自动化。提供启动 Chrome、导航页面、执行 JavaScript、截图、交互式选择 DOM 元素等功能。

**触发时机：** 用户需要进行浏览器自动化任务时：启动带远程调试的 Chrome、导航页面、在浏览器上下文中执行 JavaScript、捕获截图或交互式选择 DOM 元素。不需要 MCP 服务器。

---

#### changelog-generator
**描述：** 从 git 提交历史自动创建面向用户的变更日志。分析提交历史，分类变更，将技术性提交转换为清晰、用户友好的发布说明。

**触发时机：** 准备新版本的发布说明、创建每周或每月产品更新摘要、为客户记录变更、编写应用商店提交的变更日志条目、生成更新通知、创建内部发布文档、维护公共变更日志/产品更新页面时。

---

#### git-commit-guidelines
**描述：** 确保提交消息遵循项目约定。首先检查提交配置文件，如果没有找到则回退到 Conventional Commits 标准。提交消息应优先使用中文。

**触发时机：** ⚠️ **仅在用户明确请求 git 提交时使用**。当用户明确要求提交、说"commit"、"提交"、"git commit"等，或请求创建提交消息时使用。不要在用户未明确请求提交时使用。

---

#### using-git-worktrees
**描述：** 创建隔离的 git worktrees，具有智能目录选择和安全性验证。允许在多个分支上同时工作，无需切换。

**触发时机：** 开始需要与当前工作区隔离的功能工作，或在执行实施计划之前使用。用于创建隔离的 git worktrees，具有智能目录选择和安全性验证。

---

#### finding-duplicate-functions
**描述：** 审计代码库中的语义重复 - 执行相同功能但名称或实现不同的函数。特别适用于 LLM 生成的代码库，其中经常创建新函数而不是重用现有函数。

**触发时机：** 代码库有机增长并有多个贡献者（人类或 LLM）时、怀疑工具函数被多次重新实现时、在重大重构之前识别合并机会时、在 jscpd 已运行且语法重复已处理之后使用。

---

### 完成类技能

#### finishing-a-development-branch
**描述：** 当实现完成、所有测试通过时，指导如何完成开发工作。通过呈现结构化选项（合并、PR 或清理）来指导开发工作的完成。

**触发时机：** 实现完成、所有测试通过，需要决定如何集成工作时使用。用于通过呈现结构化选项来指导开发工作的完成。

---

#### writing-clearly-and-concisely
**描述：** 将 Strunk 的永恒写作规则应用于人类将阅读的任何散文 - 文档、提交消息、错误消息、解释、报告或 UI 文本。使你的写作更清晰、更有力、更专业。

**触发时机：** 编写人类阅读的散文时使用：文档、README 文件、技术解释、提交消息、拉取请求描述、错误消息、UI 副本、帮助文本、注释、报告、摘要或任何解释。如果你正在编写人类阅读的句子，请使用此技能。

---

### 并行处理类技能

#### dispatching-parallel-agents
**描述：** 当面对 2+ 个可以并行处理的独立任务时，为每个问题域分派一个代理，让它们并发工作。

**触发时机：** 当有多个不相关的失败（不同的测试文件、不同的子系统、不同的 bug）时，每个调查都是独立的，可以并行进行。任务之间没有共享状态或顺序依赖。

## 使用建议

1. **开始对话时**：首先使用 **using-superpowers** 技能了解如何查找和使用技能。

2. **开始新功能时**：
   - 使用 **brainstorming** 进行设计探索
   - 使用 **using-git-worktrees** 创建隔离工作区
   - 使用 **writing-plans** 编写实施计划

3. **编写代码前**：
   - JS/TS 项目：使用 **checking-auto-imports**
   - React 项目：使用 **react-19-changes**
   - Zod 验证：使用 **zod-4-changes**

4. **遇到问题时**：
   - Bug/测试失败：使用 **systematic-debugging**
   - 多个独立问题：使用 **dispatching-parallel-agents**

5. **完成工作时**：
   - 使用 **verification-before-completion** 验证
   - 使用 **requesting-code-review** 请求审查
   - 使用 **finishing-a-development-branch** 完成分支

## 注意事项

- 某些技能标记为**强制要求**（MANDATORY），必须在特定场景下使用
- 技能之间有依赖关系，请按照正确的顺序使用
- 每个技能都有详细的文档说明，使用前请仔细阅读
- 技能会不断更新，请使用最新版本
