# Skill 严谨化验收清单（适用于 tmp/**/SKILL.md）

> 用途：在每次改写后，用该清单做一次“纸上回归”，确认 Skill 触发后的行为稳定、可复核、可控上下文。

## 1. 元数据（frontmatter）

- `name` 未变（避免触发规则漂移）
- `description` 未被改成过窄/过宽（避免误触发或触发失败）

## 2. 语言与模板

- SKILL.md 的叙述性内容为英文
- 产物模板/示例模板：若原本为中文则保留中文（不强行英文化）
- 引用其他 Skill 的名称/触发语句保持原样（即便未复制到 tmp 也不是错误）

## 3. 输入/输出契约

- 明确写出 Inputs（Required），且包含“缺输入怎么处理（STOP 并澄清）”
- 明确写出 Output Requirements（硬要求），且产物结构可检查

## 4. 可执行流程

- 有 Workflow（步骤），每步都有检查点/完成定义
- 关键节点包含 Stop Conditions（不能猜，必须停并问）
- 明确 Verification（最低验证集合 + 失败处理）

## 5. 严谨性（避免“口号化”）

- 所有“完成/通过/已修复”的表述都要求证据（命令输出/可追溯引用/确定性证明）
- 发生歧义、缺复现、存在多解且影响较大时，明确要求先澄清（一次一个问题或给选项）
- 避免把实现细节写成大段可复制代码（仅允许必要的“澄清代码”）

## 6. 上下文节制

- 明确 Context Hygiene（只读最小文件集、按需加载、优先定点搜索）
- 避免在 SKILL.md 内嵌长示例/长模板（应拆到单独文件或删减）

## 7. 与其他 Skill 的衔接

- REQUIRED SUB-SKILL 的链路清楚（何时切换、为什么切换）
- 同一条链路里不存在循环或互相矛盾的要求

## 8. 纸上回归（最小演练）

对每个 Skill 做 1 次“纸上演练”，不写实现代码，只检查流程是否闭环：

- 给一段典型输入 → 能否按 Workflow 产出符合 Output Requirements 的结果？
- 中途遇到不确定信息时 → 是否会按 Stop Conditions 停并提问？
- “完成”时 → 是否能按 Verification 给出证据？
